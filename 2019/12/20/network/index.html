发表于 2019-12-20 | 更新于 2020-01-19 | 分类于 网络

从浏览器输入一个网址会发生什么？

首先，TCP/IP的体系结构将计算机网络分为四层，从下往上为网络接口层（物理层和数据链路层），网络层，传输层和应用层。
如果简述一下各层的作用，则

  1. 应用层：应用进程间的通信和交互的规则（报文）。
  2. 传输层：不同主机进程间的通信。包含两种协议：传输控制协议TCP（报文段）和用户数据报协议UDP（用户数据报）。
  3. 网络层：不同主机间的通信（IP数据报）。
  4. 网络接口层：数据链路层（帧），物理层（比特流）。

如果客户端的应用进程向服务器的应用进程传送数据，则首先从上往下（应用层->物理层）各层封装各自的控制信息再往下传送（物理层不会加上此层的控制信息），如果中途途径路由器，则需要向上剥去数据链路层的控制信息，得到网络层首部的目的地址，根据路由转发表得到相应的信息，之后再往下重新添加控制信息进行传送，直至到达服务器，此时从下往上（物理层->应用层）剥去各层的控制信息，将数据交给服务器对应的应用进程。

回到最先的问题，在我们输入一个网址后，发生的事情如下：

域名解析为IP地址

先不谈论http和https的区别。如果不输入https，则浏览器默认使用http。
如果DNS Cache和host文件中都没有记录此域名相对于的ip地址，则需要查询本地DNS服务器，当然本地DNS服务器需要途径一段路程才能到达，DNS使用的是UDP协议，默认端口号为53，则再加上一个ip地址（假设为8.8.8.8）则可以将此查询请求发送至本地DNS服务器了，因为本地DNS服务器和我们不在一个局域网内，所以会途径路由器（路由表–目的ip地址 掩码 下一跳地址），而在数据链路层上使用的是mac地址，那如何得知该路由器的mac地址呢？此时会使用到arp，如果arp缓存中没有对应的信息，则会广播一个请求询问此ip对应的mac地址是什么，对应的ip收到这条请求后会单播将mac地址回应给请求的人（此时双方都会在各自的arp缓存中加入对应的ip和mac的映射信息）。
如果本地DNS服务器中有此域名对应的ip地址则直接将结果返回。如果没有，则本地服务器需要去访问根域名服务器（全球一共13台），比如baidu.com.cn这个域名，虽然根域名服务器可能也不知道此域名对应的ip地址，但是它知道.cn的ip地址，于是本地DNS服务器又会去查询.cn对应的域名服务器，如果.cn域名服务器还是不知道，则它会告诉本地DNS服务器.com的ip地址…直到最后本地DNS服务器知道baidu.com.cn的ip地址，然后将结果返回。

TCP三次握手

在得到ip地址后，首先要建立一个连接以便以后的数据传输。

  1. 第一次：客户端发送一个SYN报文（SYN=1,seq=x），此时客户端处于SYN-SENT状态。
  2. 第二次：服务器回应一个SYN+ACK报文（SYN=1,ACK=1,seq=y,ack=x+1），此时服务器处于SYN-REVD状态。
  3. 第三次：客户端回应一个ACK报文（ACK=1,seq=x+1,ack=y+1），此时客户端处于ESTABLISHED状态，服务器收到ACK报文后，也处于ESTABLISHED状态。

为什么要三次握手？
谢希仁的《计算机网络》中的原因是：防止失效的连接请求到达服务端，服务端建立了连接，但此时客户端已经进入CLOSED状态，服务端会一直等待下去，浪费资源。
其实三次握手本质是四次握手：

  • 客户端-请求->服务端
  • 服务端-确认->客户端
  • 服务端-请求->客户端
  • 客户端-确认->服务端

其中第二和第三可以合并在一起成为一次。
同时TCP的可靠连接是靠seq来维持的，如果仅仅是两次握手，则客户端其实没有对服务端的起始序列号做确认，不能保证传输的可靠性。
半连接队列是什么？
第二次握手后，服务器将此请求连接放在一个队列中，如果迟迟得不到客户端的回应，则重传，直到重传到一定的次数，则将该请求连接从队列中删除。
三次握手过程中可以携带数据吗？
第三次握手可以携带数据，因为此时服务端和客户端都已经确认了对方的起始序列号，双方的发送和接收能力都没有问题。

关闭连接

此处涉及到TCP的四次挥手，其过程为：

  1. 客户端发出释放连接请求（FIN=1,seq=u），此时客户端处于FIN-WAIT-1状态。
  2. 服务器收到此请求后发出确认（ACK=1,ack=u+1,seq=v），此时客户端到服务端这个方向的连接关闭，此时服务器处于CLOSE-WAIT状态，而客户端接收到此确认报文后，处于FIN-WAIT-2状态。
  3. 服务器如果还有未发送完的数据，则继续发送，发送完毕后服务端给客户端发送释放连接报文（FIN=1,seq=w,ack=u+1），此时服务器处于LAST-ACK状态。
  4. 客户端给服务端发送最后的确认报文（ACK=1,ack=w+1,seq=u+1），然后进入TIME-WAIT状态，等待2MSL才进入CLOSED状态。而服务端若收到此确认报文，则直接进入CLOSED状态。

为什么要等待2MSL时间？

  1. 确保客户端发送的最后的确认报文到达服务端，如果服务端一段时间后收不到（服务端只有收到了确认报文才能进入CLOSED状态），则会重发释放连接报文，因为有了此段等待时间，客户端能收到重发的释放连接报文。
  2. 避免已失效的连接请求报文段出现在本连接中，让本连接内所产生的所有报文段都从网络中消失。

TCP可靠传输是如何实现的？

首先，先谈一下tcp和udp的主要性质。

  1. UDP：无连接；不保证可靠交付；面向报文—对应用层交付下来的报文，只添加UDP首部后就下交给网络层；没有拥塞控制（满足实时应用的需求）。
  2. TCP：建立接连；可靠交付；面向字节流（见下）；拥塞控制。
字节流特性

UDP不多详述，首先是TCP的面向字节流特性。TCP在连接的两端设有发送和接收缓存，用于临时存放发送/接收的数据。面对应用程序下交的数据，TCP仅仅把它们看成字节流，并不关心它们是什么，比如应用层放到TCP缓存中的数据太长，则TCP可以选择把它们分开再传送。这种特性结合序号便能实现可靠传输（见下），但是因为这种字节流特性，所以在写应用程序时，要把字节流正确分开（这块会涉及到socket编程，以后再说）。

可靠传输

网络层是不可靠交付，所以TCP必须保证可靠的传输，为了实现这一目的，有以下的概念：

滑动窗口
TCP滑动窗口以字节为单位，假设A发送数据，B接收数据。
发送方的窗口不能超过接收方的窗口，假设A的窗口大小为30，B的窗口大小为30。
首先A发送序号为1-10的数据，假设B收到了1-5、7-8的数据，则B交付主机，并给A发送的确认报文段中的确认号为6（期盼收到的序号），A和B的窗口同时向前移动至序号6，A在一定的时间过后没有收到确认就需要重传已发送的报文，若此次重传使得B收到序号6的数据了，则将6-8的数据交付主机并给A发送确认，此时A和B的窗口同时向前移动3个字节。

流量控制
流量控制指让发送方的发送速率不要过快，使得接收方来得及接收。
使用滑动窗口机制可以控制发送方的窗口，接收方发送的确认报文中的窗口字段可以设置发送方的窗口，从而实现对发送方的流量控制。

拥塞控制
当网络出现拥塞，则分组会丢失，此时发送方会重传导致网络更加拥塞。为了避免这一种情况，发送方在网络出现拥塞时应该降低发送的速率缓解网络拥塞。
TCP进行拥塞控制的算法有四种：慢开始、拥塞避免、快重传和快恢复。

  1. 慢开始和拥塞避免：
    一开始设置拥塞窗口为1，当收到确认后，将窗口乘以2…直至达到慢开始门限，此时进入拥塞避免，拥塞窗口以线性速度增长，如果出现了超时（收不到确认号），则将慢开始门限降至此时拥塞窗口的一半，重新执行慢开始。
  2. 快重传和快恢复：
    如果接收方收到三个重复的确认，则知道出现了丢失，则立即重传，因为此时不是网络拥塞，仅仅时丢失，所以执行快恢复，将门限和拥塞窗口都设置为当前拥塞窗口的一半。

HTTP/1.0、HTTP/1.1、HTTP/2、HTTPS？

HTTP/1.0不详述，其主要的缺点在于每个TCP连接只能发送一个请求，而TCP建立连接要三次握手，为了解决这个问题，可以设置在请求头中设置Connection为keep-alive，但是这不是标准字段，不同实现的行为可能不一致，因而引出HTTP/1.1。

HTTP/1.1

  1. 久连接，即TCP连接默认不关闭，客户端在最后一个请求，可以设置Connection为close即可关闭连接（如果客户端和服务端长时间发现对方没有活动，也可以主动关闭连接）。
  2. 在一个TCP连接中，客户端可以同时发送多个请求，但是服务端还是按照顺序一一回应。
  3. 增加了PUT、PATCH、HEAD、OPTIONS、DELETE方法。
  4. 增加Host字段，有了此字段，可以将请求发往同一台服务器上的不同网站（虚拟主机）。
    但1.1还是有缺点的，它的回应必须按照顺序，如果前一个请求的回应速度很慢，则后面的请求都得不到及时的回应，因而引出来HTTP/2。

HTTP/2

  1. HTTP/1.1的头信息是文本（ASCII编码），数据可以是文本，也可以是二进制，而HTTP/2则全部采用二进制。
  2. 如果服务器同时收到了A和B的请求，如果A请求处理很慢，则服务器可以将对A已经处理好的部分发送，然后先处理B请求。
  3. 同一个连接中连续的数据包可能属于不同的回应，为了解决此问题，每个请求/回应的数据包称为一个数据流，而每个数据流都有一个编号用以区分。客户端发出的数据流的id为奇数，服务端发出的数据流的id为偶数。客户端还可以指定数据流的优先级，优先级越高，服务器会越早回应。
  4. 头信息可以压缩发送，同时客户端和服务端同时维护一张头信息表，所有字段都会存入这张表，生成索引号，之后不发送具体字段，只发送索引号就可以了。
  5. 服务端可以主动推送给客户端数据。比如客户端解析HTML后需要再向服务端请求静态资源，此时服务器可以主动推送。

HTTPS加密（SSL/TSL）

为什么要使用HTTPS？
HTTP具有以下三个问题：

  1. 窃听
  2. 篡改
  3. 冒充
    为了解决这三个问题：
  4. 加密传播
  5. 校验
  6. 身份证书

SSL/TLS协议采用公钥加密法，客户端向服务端索要公钥，用公钥加密信息，服务端再用私钥解密。
但在这个过程中，如果有人冒充服务端的公钥，客户端得到的公钥不是真实的服务端的公钥，那么如何避免这种情况？将公钥放在数字证书里面即可，如果证书是可信的，公钥就是可信的。

数字证书/数字签名是什么？
为了验证服务端的身份—数字证书。
为了信息不被篡改—数字签名。
服务端首先会对要传输的信息（服务端的公钥及其他信息）进行hash，得到的内容称为摘要，然后对摘要使用CA的私钥进行加密，得到的内容称为数字签名，然后将数字签名和传输的信息一起放在数字证书里面传给客户端。
客户端拿到信息后，先使用CA的公钥对数字签名解密，得到摘要1，然后客户端使用相同的hash传输的信息，如果此次hash后得到的摘要2和摘要1相同，说明传输的信息没有被篡改过。

SSL/TSL协议的具体过程？

  1. 客户首先发出加密请求（支持的协议版本、随机数1、支持的加密方法等）
  2. 服务器发出回应（确认使用的协议版本，随机数2，确认使用的加密方法，服务器证书）
  3. 客户端取出证书中的公钥，如果证书没有什么问题，则客户端做出回应（随机数3–使用公钥加密、编码改变通知–确定了双方协商好的加密方法和密钥）
  4. 服务端使用自己的私钥解密第三步发出的随机数，之后发出回应（编码改变通知）
    三个随机数最终通过约定好的加密方法生成此次对话所用的“会话密钥”，之后服务端和客户端之间通信便使用此“会话密钥”进行对称加密通信。（只有第三和第四步是使用到了非对称加密，保证效率）
